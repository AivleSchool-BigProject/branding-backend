# **🟦 BrandPilot - AI 브랜드 컨설팅 서비스**
BrandPilot은 AI를 활용해 네이밍, 컨셉, 스토리, 로고를 단계적으로 생성하는 브랜드 컨설팅 플랫폼입니다.
스타트업 및 창업자를 대상으로 **브랜드 설계와 투자 게시판 기능**을 함께 제공하여, 브랜드 기획부터 투자 연결까지 지원하는 서비스를 구현했습니다.

## **📖 주요 기능**
### **1️⃣ 회원 관리**

- JWT 기반 인증/인가 처리
- 사용자별 브랜드 생성 및 관리

### **2️⃣ 인터뷰 기반 브랜드 진단**

- 사용자 입력을 기반으로 브랜드 방향성 분석
- 핵심 키워드 및 타겟 페르소나 도출

### **3️⃣ AI 브랜드 컨설팅**

- **네이밍 추천**: 사용자 니즈와 브랜드 방향성에 맞는 이름 후보 생성
- **컨셉 설계**: 선택된 네이밍과 인터뷰 내용을 반영해 브랜드 컨셉 도출
- **스토리 구성**: 누적된 Context를 기반으로 브랜드 서사와 메시지 구체화
- **로고 생성**: 앞선 단계에서 정리된 브랜드 정체성을 시각화한 로고 후보 생성

### **4️⃣  투자 게시판**

- 생성된 브랜드를 기반으로 투자/홍보 게시글 작성
- 사용자 간 브랜드 공유 및 투자 연결 지원

## **🧠 기술 블로그 기반 주요 고민 & 해결 전략**
### **Stateless AI 아키텍처의 한계 극복을 위한 'Context Versioning' 설계**

#### **Stateless AI 아키텍처의 한계 극복을 위한 'Context Versioning' 설계**

- **Problem**
    - LLM 기반 AI 서버의 Stateless 특성으로 인해 단계별(인터뷰~로고) 진행 시 이전 맥락(Context)이 유실되어 결과물의 일관성이 파괴되는 문제 발생.
- **Attempt**
    - 상태 관리 주체에 따른 3가지 아키텍처(Client-side, AI-side, BE-centralized)의 트레이드오프 분석.
    - 네트워크 오버헤드 최소화 및 데이터 정합성 보장을 위해 **백엔드 중심의 상태 관리 구조** 선택.
- **Action**
    - **Versioning 기반 State Context 저장 시스템 구축:** AI 응답에서 필수 문맥만 추출하여 RDB에 저장하고, 차기 단계 호출 시 이를 주입하는 **Bridge Logic** 구현.
    - **조회 성능 최적화:** 최신 문맥 조회를 위해 **(brand_id, step, is_active) 복합 인덱스**를 설계하여 대량 데이터 환경에서의 탐색 속도 확보.
    - **상태 전이의 원자성(Atomicity) 보장:** `@Modifying` 쿼리를 통한 일괄 업데이트 로직을 적용하여 단계 전환 중 발생할 수 있는 데이터 불일치 차단.
- **Result**
    - 서버 장애나 재시작 상황에서도 **서비스 복구 능력** 확보 및 AI 서버와 백엔드 간 명확한 관심사 분리

#### **API 구간별 프로파일링을 통한 성능 병목 지점 특정 및 최적화 실험**

- **Problem**
    - 서비스 단계 진행에 따른 응답 시간의 선형적 증가로 UX 저하 및 사용자 이탈 위기 발생.
- **Attempt**
    - **구간별 Latency Tracking:** AOP 기반의 커스텀 로깅을 구현하여 전송/추론/가공 각 구간의 지연 시간을 정밀 측정.
    - **가설 설정 및 검증:** "페이로드 크기가 병목의 핵심"이라는 가설 하에 데이터 크기를 **46.3% 축소**하는 대조 실험 수행.
- **Action**
    - **데이터 기반 병목 확증:** 실험 결과 전송량 감소에도 응답 속도는 개선되지 않음을 확인하여, 병목의 실체가 전송량이 아닌 'AI 모델의 토큰 생성 비용 및 추론 복잡도'에 있음을 기술적으로 증명.
    - **전략 수정:** 무의미한 페이로드 축소 대신, 외부 API 호출 구간에 WebClient를 도입하여 I/O 바운드 구간의 스레드 효율성을 극대화하는 방향으로 선회.
- **Result**
    - 잘못된 가설에 근거한 리소스 낭비를 차단하여 **개발 사이클 1주 이상 단축**.
    - 동기 방식 대비 **TPS(초당 처리량) 약 20% 이상 개선** 및 시스템 가용성 확보.

#### FSM 기반 도메인 상태 제어 및 비즈니스 무결성 확보

- **Problem**
    - 비정상적인 URL 접근(단계 건너뛰기) 등 정해진 워크플로우를 우회하는 요청 시 데이터 정합성이 파손되는 보안 취약점 존재.
- **Attempt**
    - 복잡한 IF-ELSE 분기 처리와 명시적 상태 제어(**FSM**) 모델 간의 가시성 및 유지보수 확장성 비교 분석.
- **Action**
    - **FSM 기반 상태 전이 엔진 설계:** 엔티티 내부에 상태 전이 로직을 캡슐화하여 서비스 레이어의 방어 로직을 강화하고 비정상 접근을 원천 차단.
    - **상태 전이 가시성 확보:** 상태 규칙을 명시적으로 정의하여 코드 복잡도를 **약 30% 감소**시키고 유지보수 안정성 제고.
- **Result**
    - 비정상적인 단계 진입 시도를 애플리케이션 레이어에서 **100% 차단**하여 서비스 무결성 완수.

#### 대규모 조회 성능 개선을 위한 DB 최적화 전략

- **problem**
    - 마이페이지 등 대량의 브랜드 리스트 조회 시 N+1 문제로 인한 지연 발생 및 데이터 증가에 따른 성능 저하 우려.
- **Attempt**
    - EXPLAIN 분석을 통해 쿼리 실행 계획을 진단하고, Full Table Scan 구간에 대한 인덱스 전략 수립.
- **Action**
    - **Memory Aggregation 전략 적용:** `findByIn` 쿼리로 연관 데이터를 일괄 조회 후, Map 자료구조를 활용해 메모리 내에서 매핑하는 구조 도입
    - **복합 인덱스 설계:** (brand_id, step, is_active) 인덱스를 생성하여 데이터 탐색 범위를 최적화(Index Range Scan).
- **Result**
    - 쿼리 발생량을 98% 줄여 대규모 트래픽 환경에서도 일관된 응답 속도 확보 및 DB 가용성 극대화.

#### 클라우드 리소스 최적화를 위한 연쇄 삭제 파이프라인 구축

- problem
    - DB 레코드는 삭제되나 외부 스토리지(S3) 파일이 잔류하는 고아 객체(Orphan Object) 발생으로 인한 스토리지 비용 누수 및 불일치 위험.
- Attempt
    - 스토리지 관리 효율화를 위해 배치(Batch)기반 사후 정리 방식과 실시간 연쇄 삭제 방식의 트레이드 오프 검토.
- Action
    - **Atomic Cleanup 파이프라인 구현**: DB 트랜잭션 내 S3 API 연동을 통해 레코드 삭제 시 스토리지 파일까지 원자적으로 제거하는 로직 구축.
    - **스토리지 정합성 보장**: RDB와 외부 스토리지 간의 ㄷ[이터 일관성을 유지하여 운영 환경에서의 수동 관리 리소스 제거.
- Result
    - 불필요한 스토리지 비용 발생을 사전에 차단하고, 실제 서비스 데이터와 스토리지 자산 간의 1:1 정합성 확보

#### **Docker Orchestration 및 이미지 기반 CI/CD 자동화 구축**

- **Problem**
    - Spring Boot, FastAPI, React 등 서로 다른 실행 환경의 의존성 관리 복잡도 증가.
- **Attempt**
    - 실행 환경의 일관성 유지와 손쉬운 롤백을 위해, 소스 코드 직접 배포 방식 대신 **컨테이너 이미지 기반 배포(Docker)** 전략 채택.
- **Action**
    - **Multi-Container 통합 및 내부 네트워크 설계:** 3개 서비스를 하나의 가상 네트워크로 격리하고, **컨테이너 이름 기반의 내부 DNS 통신** 구조를 설계하여 서비스 간 연결 엔드포인트 관리 복잡도 해소.
    - **GitHub Actions 기반 CI/CD 파이프라인 구축:** 코드 Push 시 Docker 이미지를 자동 빌드 및 Docker Hub에 푸시하고, 운영 서버(EC2)에서 최신 이미지를 Pull 하여 무중단에 가까운 재기동을 수행하는 워크플로우 자동화.
- **Result**
    - 서버 환경 의존성을 완벽히 제거하고, 배포 실수 가능성을 차단하여 운영 서버를 '실행 전용 환경'으로 고정함으로써 안정성 확보.

#### **계층별 보안(Layered Security) 아키텍처 및 인프라 설계**

- **Problem**
    - 각 서비스 포트 노출에 따른 보안 취약점과 전송 데이터 평문 전송으로 인한 가로채기 위험 존재.
- **Action**
    - **단일 진입점(Nginx Gateway) 설계**: 외부 포트는 오직 80/443만 개방하고, 내부 서비스(Spring, FastAPI)는 Docker 내부 네트워크에서만 통신하도록 접근 제어 강화.
    - **전송 구간 암호화(HTTPS)**: Let's Encrypt 및 Certbot을 통해 SSL 인증서를 적용하고, 80 포트 접속 시 HTTPS로 강제 리다이렉트 처리.
    - **인프라 격리**: RDS의 Public Access를 차단하고, **AWS 보안 그룹(Security Group)** 설정을 통해 오직 백엔드 EC2 인스턴스의 3306 포트 접근만 허용하도록 네트워크 레벨에서 DB 격리.
- **Result**
    - 외부 위협으로부터 핵심 리소스(DB, API)를 격리하고, 안정적인 전송 보안과 네트워크 보안을 동시에 충족하는 인프라 구조 완성.
